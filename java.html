Project Setup

Initialize your Spring Boot project and understand the structure

What is Spring Boot?

Spring Boot is a Java framework that makes it easy to create stand-alone, production-grade applications. Think of it as a toolkit that handles the boilerplate (repetitive setup code) so you can focus on your business logic — the code that actually does what your app needs to do.

Why use it? Without Spring Boot, setting up a Java web application means manually configuring servers, database connections, security, and dozens of other things. Spring Boot does all of this automatically using sensible defaults — a concept called "convention over configuration."

02

REST API Basics & CRUD

Build your first API endpoints with Controller → Service → Repository

Understanding REST

REST (Representational State Transfer) is a way to design web APIs. Your API is a set of URLs (called endpoints) that clients (web browsers, mobile apps) can call to read, create, update, or delete data. Each HTTP method has a purpose:

GET = Read data  |  POST = Create data  |  PUT = Update data  |  DELETE = Remove data

Entity (Database Model)

An entity represents a table in your database. Each instance of the class is a row. Lombok annotations like @Data automatically generate getters, setters, and toString methods — saving you from writing boilerplate code.

Item.java — Entity 
package com.example.myapp.model.entity;

import jakarta.persistence.*;
import lombok.*;
import java.time.LocalDateTime;

@Entity
@Table(name = "items")
@Data                    // Generates getters, setters, toString, equals, hashCode
@NoArgsConstructor       // Generates no-argument constructor
@AllArgsConstructor      // Generates all-argument constructor
@Builder                 // Enables the Builder pattern (covered later)
public class Item {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 255)
    private String name;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(nullable = false)
    private Double price;

    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at")
    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
        this.updatedAt = LocalDateTime.now();
    }

    @PreUpdate
    protected void onUpdate() {
        this.updatedAt = LocalDateTime.now();
    }
}

DTOs (Data Transfer Objects)

DTOs are simple classes that carry data between your API and the outside world. Why not just use the entity? Because entities may contain sensitive fields (like passwords), database-specific annotations, or fields that shouldn't be exposed. DTOs give you control over what goes in and out of your API.

ItemRequest.java — DTO for incoming data 
package com.example.myapp.model.dto;

import jakarta.validation.constraints.*;
import lombok.Data;

@Data
public class ItemRequest {

    @NotBlank(message = "Name is required")
    @Size(max = 255, message = "Name must be under 255 characters")
    private String name;

    private String description;

    @NotNull(message = "Price is required")
    @Positive(message = "Price must be positive")
    private Double price;
}
ItemResponse.java — DTO for outgoing data 
package com.example.myapp.model.dto;

import lombok.Builder;
import lombok.Data;
import java.time.LocalDateTime;

@Data
@Builder
public class ItemResponse {
    private Long id;
    private String name;
    private String description;
    private Double price;
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
}

Repository

The repository is your database access layer. Spring Data JPA gives you CRUD operations for free — just extend the interface and Spring generates the implementation at runtime.

ItemRepository.java 
package com.example.myapp.repository;

import com.example.myapp.model.entity.Item;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;
import java.util.List;

@Repository
public interface ItemRepository extends JpaRepository<Item, Long> {

    // Spring generates the SQL query from the method name!
    List<Item> findByNameContainingIgnoreCase(String name);

    List<Item> findByPriceLessThan(Double price);
}

Service Layer

The service layer contains your business logic. It sits between the controller and repository. Why a separate layer? It keeps your controller thin (just handling HTTP) and makes business logic easy to test independently.

ItemService.java 
package com.example.myapp.service;

import com.example.myapp.model.dto.*;
import java.util.List;

public interface ItemService {
    ItemResponse create(ItemRequest request);
    ItemResponse getById(Long id);
    List<ItemResponse> getAll();
    ItemResponse update(Long id, ItemRequest request);
    void delete(Long id);
}
ItemServiceImpl.java 
package com.example.myapp.service.impl;

import com.example.myapp.exception.ResourceNotFoundException;
import com.example.myapp.model.dto.*;
import com.example.myapp.model.entity.Item;
import com.example.myapp.repository.ItemRepository;
import com.example.myapp.service.ItemService;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

@Service
@RequiredArgsConstructor  // Lombok generates constructor for final fields
@Transactional
public class ItemServiceImpl implements ItemService {

    private final ItemRepository itemRepository;

    @Override
    public ItemResponse create(ItemRequest request) {
        Item item = Item.builder()
                .name(request.getName())
                .description(request.getDescription())
                .price(request.getPrice())
                .build();
        Item saved = itemRepository.save(item);
        return mapToResponse(saved);
    }

    @Override
    @Transactional(readOnly = true)
    public ItemResponse getById(Long id) {
        Item item = itemRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException(
                    "Item not found with id: " + id));
        return mapToResponse(item);
    }

    @Override
    @Transactional(readOnly = true)
    public List<ItemResponse> getAll() {
        return itemRepository.findAll().stream()
                .map(this::mapToResponse)
                .toList();
    }

    @Override
    public ItemResponse update(Long id, ItemRequest request) {
        Item item = itemRepository.findById(id)
                .orElseThrow(() -> new ResourceNotFoundException(
                    "Item not found with id: " + id));
        item.setName(request.getName());
        item.setDescription(request.getDescription());
        item.setPrice(request.getPrice());
        Item updated = itemRepository.save(item);
        return mapToResponse(updated);
    }

    @Override
    public void delete(Long id) {
        if (!itemRepository.existsById(id)) {
            throw new ResourceNotFoundException(
                "Item not found with id: " + id);
        }
        itemRepository.deleteById(id);
    }

    private ItemResponse mapToResponse(Item item) {
        return ItemResponse.builder()
                .id(item.getId())
                .name(item.getName())
                .description(item.getDescription())
                .price(item.getPrice())
                .createdAt(item.getCreatedAt())
                .updatedAt(item.getUpdatedAt())
                .build();
    }
}

Controller

The controller is the entry point for HTTP requests. It defines your API endpoints, calls the service layer, and returns responses. Keep controllers thin — they should only handle HTTP concerns (status codes, request/response mapping).

ItemController.java 
package com.example.myapp.controller;

import com.example.myapp.model.dto.*;
import com.example.myapp.service.ItemService;
import io.swagger.v3.oas.annotations.Operation;
import io.swagger.v3.oas.annotations.tags.Tag;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.util.List;

@RestController
@RequestMapping("/api/v1/items")
@RequiredArgsConstructor
@Tag(name = "Items", description = "Item management endpoints")
public class ItemController {

    private final ItemService itemService;

    @PostMapping
    @Operation(summary = "Create a new item")
    public ResponseEntity<ItemResponse> create(
            @Valid @RequestBody ItemRequest request) {
        return ResponseEntity.status(HttpStatus.CREATED)
                .body(itemService.create(request));
    }

    @GetMapping("/{id}")
    @Operation(summary = "Get item by ID")
    public ResponseEntity<ItemResponse> getById(@PathVariable Long id) {
        return ResponseEntity.ok(itemService.getById(id));
    }

    @GetMapping
    @Operation(summary = "Get all items")
    public ResponseEntity<List<ItemResponse>> getAll() {
        return ResponseEntity.ok(itemService.getAll());
    }

    @PutMapping("/{id}")
    @Operation(summary = "Update an item")
    public ResponseEntity<ItemResponse> update(
            @PathVariable Long id,
            @Valid @RequestBody ItemRequest request) {
        return ResponseEntity.ok(itemService.update(id, request));
    }

    @DeleteMapping("/{id}")
    @Operation(summary = "Delete an item")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        itemService.delete(id);
        return ResponseEntity.noContent().build();
    }
}

Validation & Exception Handling

Validate input and handle errors gracefully

Why Validation Matters

Never trust user input. Validation ensures that data meets your requirements before it hits your business logic or database. Spring Boot's @Valid annotation combined with Jakarta Bean Validation annotations makes this easy.

Common validation annotations: @NotBlank (not empty), @NotNull, @Size, @Email, @Positive, @Pattern (regex match). We already saw these on our DTO — now let's handle what happens when validation fails.

Global Exception Handler

Instead of handling exceptions in every controller, a global exception handler catches errors from anywhere in your app and returns clean, consistent error responses. This is done with @RestControllerAdvice.

GlobalExceptionHandler.java 
package com.example.myapp.exception;

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.*;
import java.time.LocalDateTime;
import java.util.*;

@RestControllerAdvice
public class GlobalExceptionHandler {

    // Handles validation errors (e.g., @NotBlank fails)
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new LinkedHashMap<>();
        ex.getBindingResult().getFieldErrors().forEach(err ->
            errors.put(err.getField(), err.getDefaultMessage()));

        Map<String, Object> body = new LinkedHashMap<>();
        body.put("timestamp", LocalDateTime.now());
        body.put("status", 400);
        body.put("error", "Validation Failed");
        body.put("details", errors);

        return ResponseEntity.badRequest().body(body);
    }

    // Handles "resource not found" errors
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<Map<String, Object>> handleNotFound(
            ResourceNotFoundException ex) {
        Map<String, Object> body = new LinkedHashMap<>();
        body.put("timestamp", LocalDateTime.now());
        body.put("status", 404);
        body.put("error", "Not Found");
        body.put("message", ex.getMessage());

        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(body);
    }

    // Catches any unhandled exceptions
    @ExceptionHandler(Exception.class)
    public ResponseEntity<Map<String, Object>> handleGeneral(
            Exception ex) {
        Map<String, Object> body = new LinkedHashMap<>();
        body.put("timestamp", LocalDateTime.now());
        body.put("status", 500);
        body.put("error", "Internal Server Error");
        body.put("message", "An unexpected error occurred");
        // Don't expose stack traces in production!
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(body);
    }
}
ResourceNotFoundException.java 
package com.example.myapp.exception;

public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}
⚠️ Common Mistake: Never expose stack traces or internal error details in production API responses. Attackers can use this information to find vulnerabilities. Always log the full error server-side but return a generic message to clients.

Database with Liquibase & PostGIS

Manage database migrations and add geographic support

What is Liquibase?

Liquibase is a database migration tool. It tracks changes to your database schema (tables, columns, indexes) using versioned "changelog" files. Think of it as Git for your database — every change is recorded, ordered, and reproducible.

Why not let Hibernate auto-create tables? Hibernate's ddl-auto: update can create and alter tables, but it can't rename columns, migrate data, or guarantee consistency across environments. Liquibase gives you full control and an audit trail.

db/changelog/db.changelog-master.yaml 
databaseChangeLog:
  - include:
      file: db/changelog/001-create-items-table.yaml
  - include:
      file: db/changelog/002-add-postgis-support.yaml

PostGIS: Geographic Data

PostGIS is a PostgreSQL extension that adds support for geographic objects — points, lines, polygons, etc. Combined with Hibernate Spatial, you can store locations, areas, and routes directly in your database and query them spatially (e.g., "find all items within 5 km of this point").

db/changelog/002-add-postgis-support.yaml 
databaseChangeLog:
  - changeSet:
      id: 002-enable-postgis
      author: developer
      changes:
        - sql:
            sql: CREATE EXTENSION IF NOT EXISTS postgis;
  - changeSet:
      id: 002-add-location-columns
      author: developer
      changes:

            tableName: items
            columns:
              - column:
                  name: location
                  type: GEOMETRY(Point, 4326)

                  name: boundary

        - createIndex:
            tableName: items
            indexName: idx_items_location

PostGIS Entity Example

GeoItem.java — Entity with GIS columns 
package com.example.myapp.model.entity;

import jakarta.persistence.*;
import lombok.*;
import org.locationtech.jts.geom.Point;
import org.locationtech.jts.geom.Polygon;

@Entity
@Table(name = "items")
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class GeoItem {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    // A geographic point (latitude/longitude)
    // SRID 4326 = WGS 84, the standard GPS coordinate system
    @Column(columnDefinition = "GEOMETRY(Point, 4326)")
    private Point location;

    // A geographic polygon (area boundary)
    @Column(columnDefinition = "GEOMETRY(Polygon, 4326)")
    private Polygon boundary;
}

GIS API for OpenLayers

OpenLayers is a JavaScript library for interactive maps. It typically consumes GeoJSON data. Here's how to return data in a format OpenLayers can display directly:

GeoJSON response for OpenLayers 
// Controller endpoint returning GeoJSON
@GetMapping("/api/v1/items/geojson")
public ResponseEntity<Map<String, Object>> getGeoJson() {
    List<GeoItem> items = geoItemRepository.findAll();

    List<Map<String, Object>> features = items.stream()
        .filter(i -> i.getLocation() != null)
        .map(item -> Map.of(
            "type", "Feature",
            "geometry", Map.of(
                "type", "Point",
                "coordinates", new double[]{
                    item.getLocation().getX(),  // longitude
                    item.getLocation().getY()   // latitude
                }
            ),
            "properties", Map.of(
                "id", item.getId(),
                "name", item.getName()
            )
        )).toList();

    Map<String, Object> geoJson = Map.of(
        "type", "FeatureCollection",
        "features", features
    );
    return ResponseEntity.ok(geoJson);
}

Security: OIDC, Keycloak & Roles

Secure your API with OAuth2 and role-based access control

Understanding OAuth2 & OIDC

OAuth2 is an authorization framework — it lets users grant limited access to their resources without sharing passwords. OIDC (OpenID Connect) adds an identity layer on top of OAuth2, so you can also verify who the user is.

Keycloak is an open-source Identity Provider (IdP) that handles user registration, login, password resets, and token issuance. Your Spring Boot app becomes a Resource Server that validates tokens issued by Keycloak.

application-secure.yml — Keycloak config 
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: ${KEYCLOAK_URL:http://localhost:8180}/realms/myapp
          jwk-set-uri: ${KEYCLOAK_URL:http://localhost:8180}/realms/myapp/protocol/openid-connect/certs

# Custom property for role claim extraction
app:
  security:
    roles-claim: realm_access.roles
SecurityConfig.java — with role-based access 
package com.example.myapp.config;

import org.springframework.context.annotation.*;
import org.springframework.security.config.annotation.method.configuration.EnableMethodSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.oauth2.server.resource.authentication.JwtAuthenticationConverter;
import org.springframework.security.oauth2.server.resource.authentication.JwtGrantedAuthoritiesConverter;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
@EnableWebSecurity
@EnableMethodSecurity  // Enables @PreAuthorize on methods
@Profile("!nosecurity")  // This config is DISABLED when 'nosecurity' profile is active
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())  // Disable CSRF for stateless APIs
            .sessionManagement(s -> s.sessionCreationPolicy(
                SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                // Public endpoints
                .requestMatchers("/actuator/health").permitAll()
                .requestMatchers("/swagger-ui/**", "/v3/api-docs/**").permitAll()
                // Role-restricted endpoints
                .requestMatchers("/api/v1/admin/**").hasRole("ADMIN")
                // Everything else requires authentication
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    .jwtAuthenticationConverter(jwtAuthConverter())
                )
            );
        return http.build();
    }

    // Extracts roles from Keycloak JWT token
    private JwtAuthenticationConverter jwtAuthConverter() {
        JwtGrantedAuthoritiesConverter grantedAuthorities =
            new JwtGrantedAuthoritiesConverter();
        grantedAuthorities.setAuthorityPrefix("ROLE_");
        grantedAuthorities.setAuthoritiesClaimName(
            "realm_access.roles");

        JwtAuthenticationConverter converter =
            new JwtAuthenticationConverter();
        converter.setJwtGrantedAuthoritiesConverter(grantedAuthorities);
        return converter;
    }
}

Role-Based Access on Methods

Using @PreAuthorize for role checks 
import org.springframework.security.access.prepost.PreAuthorize;

@RestController
@RequestMapping("/api/v1/admin")
public class AdminController {

    @GetMapping("/users")
    @PreAuthorize("hasRole('ADMIN')")
    public ResponseEntity<List<UserResponse>> getAllUsers() {
        // Only users with ADMIN role can access this
        return ResponseEntity.ok(userService.getAll());
    }

    @DeleteMapping("/items/{id}")
    @PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")
    public ResponseEntity<Void> deleteItem(@PathVariable Long id) {
        // ADMIN or MANAGER can delete
        itemService.delete(id);
        return ResponseEntity.noContent().build();
    }
}

Disable Security for Local Development

During local development, you often want to test your API without dealing with tokens. The nosecurity profile disables all security:

NoSecurityConfig.java 
@Configuration
@EnableWebSecurity
@Profile("nosecurity")  // Only active with: SPRING_PROFILES_ACTIVE=nosecurity
public class NoSecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .anyRequest().permitAll()  // Allow everything
            );
        return http.build();
    }
}

// Run with: java -jar myapp.jar --spring.profiles.active=nosecurity
// Or set env: SPRING_PROFILES_ACTIVE=nosecurity
⚠️ Important: NEVER deploy the nosecurity profile to production. Use environment variables to control which profile is active in each environment.

Stub Mode for Testing

Return static responses for frontend development

Why Stub Mode?

When frontend developers are building the UI, they need predictable API responses without a real database. A stub service returns hard-coded (static) data so the frontend can be developed and tested independently. This is especially useful when the backend is still being built.

ItemServiceStub.java 
package com.example.myapp.service.impl;

import com.example.myapp.model.dto.*;
import com.example.myapp.service.ItemService;
import org.springframework.context.annotation.Profile;
import org.springframework.stereotype.Service;
import java.time.LocalDateTime;
import java.util.*;

@Service
@Profile("stub")  // Only active with: SPRING_PROFILES_ACTIVE=stub

S3: File Storage

Amazon S3 (Simple Storage Service) is cloud file storage. Use it for user uploads, images, documents, backups — anything you don't want on your server's disk. SQS (Simple Queue Service) is a message queue for async communication between services.

            // env vars → system props → IAM role → profile

                DefaultCredentialsProvider.create())

S3Service.java — File upload/download 
@Service
@RequiredArgsConstructor
public class S3Service {

    private final S3Client s3Client;

    @Value("${aws.s3.bucket}")
    private String bucket;

    public String uploadFile(String key, byte[] content,
                             String contentType) {
        s3Client.putObject(
            PutObjectRequest.builder()
                .bucket(bucket)
                .key(key)
                .contentType(contentType)
                .build(),
            RequestBody.fromBytes(content)
        );
        return String.format("https://%s.s3.amazonaws.com/%s",
                             bucket, key);
    }

    public byte[] downloadFile(String key) {
        ResponseBytes<GetObjectResponse> response =
            s3Client.getObjectAsBytes(
                GetObjectRequest.builder()
                    .bucket(bucket).key(key).build());
        return response.asByteArray();
    }
}
SqsService.java — Send/receive messages 
@Service
@RequiredArgsConstructor
@Slf4j  // Lombok annotation for logging
public class SqsService {

    private final SqsClient sqsClient;

    @Value("${aws.sqs.queue-url}")
    private String queueUrl;

    public void sendMessage(String messageBody) {
        sqsClient.sendMessage(
            SendMessageRequest.builder()
                .queueUrl(queueUrl)
                .messageBody(messageBody)
                .build()
        );
        log.info("Message sent to SQS: {}", messageBody);
    }

    public List<Message> receiveMessages(int maxMessages) {
        ReceiveMessageResponse response =
            sqsClient.receiveMessage(
                ReceiveMessageRequest.builder()
                    .queueUrl(queueUrl)
                    .maxNumberOfMessages(maxMessages)
                    .waitTimeSeconds(10)  // Long polling
                    .build()
            );
        return response.messages();
    }
}

Temporal.io Workflows

Durable, reliable workflow orchestration

What is Temporal?

Temporal is a workflow engine. It runs long-running, reliable processes that survive crashes, restarts, and network failures. Think of it as a "never-forget" task runner. Use it for: order processing, data pipelines, scheduled jobs, multi-step approval workflows, and anything that must complete even if your server restarts.

OrderWorkflow.java — Workflow interface 
package com.example.myapp.temporal;

import io.temporal.workflow.*;

@WorkflowInterface
public interface OrderWorkflow {

    @WorkflowMethod
    String processOrder(String orderId);
}

// Activity interface — the actual work steps
@ActivityInterface
public interface OrderActivities {

    String validateOrder(String orderId);
    String chargePayment(String orderId);
    String shipOrder(String orderId);
    void sendConfirmation(String orderId);
}
OrderWorkflowImpl.java — Workflow implementation 
public class OrderWorkflowImpl implements OrderWorkflow {

    private final OrderActivities activities =
        Workflow.newActivityStub(
            OrderActivities.class,
            ActivityOptions.newBuilder()
                .setStartToCloseTimeout(Duration.ofMinutes(5))
                .setRetryOptions(RetryOptions.newBuilder()
                    .setMaximumAttempts(3)
                    .build())
                .build()
        );

    @Override
    public String processOrder(String orderId) {
        // Each step is durable — if the server crashes
        // between steps, Temporal resumes from where it left off
        String validation = activities.validateOrder(orderId);
        String payment = activities.chargePayment(orderId);
        String shipping = activities.shipOrder(orderId);
        activities.sendConfirmation(orderId);
        return "Order " + orderId + " completed!";
    }
}
TemporalConfig.java — Worker setup 
@Configuration
public class TemporalConfig {

    @Value("${temporal.server:localhost:7233}")
    private String temporalServer;

    @Bean
    public WorkflowClient workflowClient() {
        WorkflowServiceStubs service = WorkflowServiceStubs.newServiceStubs(
            WorkflowServiceStubsOptions.newBuilder()
                .setTarget(temporalServer)
                .build()
        );
        return WorkflowClient.newInstance(service);
    }

    @Bean
    public WorkerFactory workerFactory(WorkflowClient client) {
        WorkerFactory factory = WorkerFactory.newInstance(client);
        Worker worker = factory.newWorker("ORDER_TASK_QUEUE");
        worker.registerWorkflowImplementationTypes(
            OrderWorkflowImpl.class);
        worker.registerActivitiesImplementations(
            new OrderActivitiesImpl());
        factory.start();
        return factory;
    }
}

Testing with JUnit & Mockito

Write reliable unit tests for your application

Why Test?

Tests are your safety net. They catch bugs before users do, give you confidence to refactor, and serve as living documentation of how your code should behave. JUnit is the test framework; Mockito creates fake ("mock") objects so you can test one layer in isolation.

ItemServiceImplTest.java 
package com.example.myapp.service.impl;

import com.example.myapp.exception.ResourceNotFoundException;
import com.example.myapp.model.dto.*;
import com.example.myapp.model.entity.Item;
import com.example.myapp.repository.ItemRepository;
import org.junit.jupiter.api.*;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.*;
import org.mockito.junit.jupiter.MockitoExtension;
import java.util.Optional;
import static org.assertj.core.api.Assertions.*;
import static org.mockito.ArgumentMatchers.any;
import static org.mockito.Mockito.*;

@ExtendWith(MockitoExtension.class)  // Enable Mockito
class ItemServiceImplTest {

    @Mock  // Create a fake repository
    private ItemRepository itemRepository;

    @InjectMocks  // Inject the mock into the real service
    private ItemServiceImpl itemService;

    private Item sampleItem;
    private ItemRequest sampleRequest;

    @BeforeEach
    void setUp() {
        sampleItem = Item.builder()
            .id(1L).name("Test Item")
            .description("A test").price(10.0).build();

        sampleRequest = new ItemRequest();
        sampleRequest.setName("Test Item");
        sampleRequest.setDescription("A test");
        sampleRequest.setPrice(10.0);
    }

    @Test
    @DisplayName("Should create item successfully")
    void createItem_Success() {
        // Arrange: tell the mock what to return
        when(itemRepository.save(any(Item.class)))
            .thenReturn(sampleItem);

        // Act: call the method under test
        ItemResponse result = itemService.create(sampleRequest);

        // Assert: verify the result
        assertThat(result.getName()).isEqualTo("Test Item");
        assertThat(result.getPrice()).isEqualTo(10.0);
        verify(itemRepository, times(1)).save(any(Item.class));
    }

    @Test
    @DisplayName("Should throw when item not found")
    void getById_NotFound_ThrowsException() {
        when(itemRepository.findById(99L))
            .thenReturn(Optional.empty());

        assertThatThrownBy(() -> itemService.getById(99L))
            .isInstanceOf(ResourceNotFoundException.class)
            .hasMessageContaining("not found");
    }

    @Test
    @DisplayName("Should return item when found")
    void getById_Found_ReturnsItem() {
        when(itemRepository.findById(1L))
            .thenReturn(Optional.of(sampleItem));

        ItemResponse result = itemService.getById(1L);

        assertThat(result).isNotNull();
        assertThat(result.getId()).isEqualTo(1L);
    }
}
💡 Testing Best Practices: Follow the AAA pattern — Arrange (set up data), Act (call the method), Assert (check the result). Name tests descriptively. Test both happy paths and error cases. Aim for 80%+ code coverage on business logic.

Design Patterns & Best Practices

Write clean, maintainable, professional code

Design Patterns Explained Simply

Design patterns are proven solutions to common programming problems. They're like recipes — once you know them, you can apply them over and over. Here are the most important ones for Spring Boot:

1. Controller → Service → Repository

This is the layered architecture we've been using throughout this tutorial. Each layer has one job: Controllers handle HTTP, Services handle business logic, Repositories handle data. This separation makes your code testable and maintainable.

2. Dependency Injection (DI)

Instead of creating objects yourself (new ItemService()), Spring injects them for you. This is why we use @RequiredArgsConstructor and final fields. DI makes testing easy (inject mocks instead of real objects) and reduces coupling between classes.

3. Builder Pattern

Creates objects step-by-step instead of using massive constructors. Lombok's @Builder generates this for you:

Builder Pattern example 
// Without builder — hard to read, easy to mix up parameters
Item item = new Item(null, "Widget", "Desc", 29.99, null, null);

// With builder — clear and readable
Item item = Item.builder()
    .name("Widget")
    .description("Desc")
    .price(29.99)
    .build();

4. Strategy Pattern

Defines a family of interchangeable algorithms. Our stub vs real service is an example — same interface, different implementations swapped based on configuration:

Strategy Pattern — Payment example 
// Strategy interface
public interface PaymentStrategy {
    PaymentResult process(PaymentRequest request);
}

@Service @Profile("!test")
public class StripePaymentStrategy implements PaymentStrategy {
    public PaymentResult process(PaymentRequest req) {
        // Real Stripe API call
    }
}

@Service @Profile("test")
public class MockPaymentStrategy implements PaymentStrategy {
    public PaymentResult process(PaymentRequest req) {
        return PaymentResult.success(); // Always succeeds
    }
}

// Spring injects the right one based on active profile!

5. Factory Pattern

Creates objects without exposing the creation logic. Useful when the type of object depends on input:

Factory Pattern 
@Component
public class NotificationFactory {

    public NotificationSender create(String type) {
        return switch (type) {
            case "email" -> new EmailSender();
            case "sms"   -> new SmsSender();
            case "push"  -> new PushSender();
            default -> throw new IllegalArgumentException(
                "Unknown notification type: " + type);
        };
    }
}

6. Adapter Pattern

Makes incompatible interfaces work together. Common when integrating with external APIs — you wrap their response format into your own DTO format, shielding your code from external changes.

Best Practices Checklist

✅ DTO ↔ Entity separation: Never expose entities directly in your API

✅ Service layer: Keep controllers thin, put logic in services

✅ Validation: Validate all input at the API boundary

✅ Exception handling: Use global exception handlers, never expose internals

✅ Secrets: Use environment variables or a secrets manager, never hard-code

✅ Logging: Use SLF4J/Lombok's @Slf4j, log at appropriate levels

✅ Profiles: Use Spring profiles for environment-specific config

✅ API versioning: Prefix endpoints with /api/v1/

✅ Testing: Write unit tests for services, integration tests for controllers

✅ Documentation: Use Swagger annotations on every endpoint

✅ Clean code: Small methods, descriptive names, single responsibility

✅ Database: Use Liquibase for migrations, never rely on Hibernate auto-DDL in production
